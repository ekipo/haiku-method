Arrow-key roving tabindex (FB-65)

Every stage node inside a <nav aria-label="Stage progress"> is reachable via arrow keys, including upcoming / never-visited stages (tabindex="-1" in the Tab order). This satisfies WCAG 2.1.1 Keyboard — even read-only informational affordances must be reachable. It also satisfies WCAG 1.4.13 Content on Hover — focusing a future stage reveals the same tooltip that appears on hover, so the tooltip content is not hover-only.

Key Action Scope

ArrowRight / ArrowDownMove focus to next stage node (wraps to first after last)Any stage node
ArrowLeft / ArrowUpMove focus to previous stage node (wraps to last before first)Any stage node
HomeMove focus to first stage nodeAny stage node

Roving-tabindex algorithm

Only ONE stage node is a Tab stop at a time. That node has tabindex="0"; all others have tabindex="-1". Arrow keys shift the tabindex="0" to the next/prev node and call focus() on the newly-roved node. This pattern is WAI-ARIA Authoring Practices canonical for toolbars, tablists, and stage-like navigations. 

// Pseudocode — dev stage implements this in React as StageProgressStrip.tsx
// DOM shape: <nav aria-label="Stage progress"><div class="stage-node" ...></div> x N </nav>

function handleArrowNav(ev: KeyboardEvent) {
  const nav = (ev.currentTarget as HTMLElement).closest('nav[aria-label*="Stage progress"]');
  if (!nav) return;
  const nodes = Array.from(nav.querySelectorAll('.stage-node'));
  const currentIdx = nodes.indexOf(document.activeElement as HTMLElement);
  if (currentIdx === -1) return;

  let nextIdx = currentIdx;
  switch (ev.key) {
    case 'ArrowRight':
    case 'ArrowDown':
      nextIdx = (currentIdx + 1) % nodes.length;
      break;
    case 'ArrowLeft':
    case 'ArrowUp':
      nextIdx = (currentIdx - 1 + nodes.length) % nodes.length;
      break;
    case 'Home':
      nextIdx = 0;
      break;
    case 'End':
      nextIdx = nodes.length - 1;
      break;
    case 'Escape':
      // Dismiss any visible tooltip without moving focus
      nodes[currentIdx].blur();
      nodes[currentIdx].focus({ preventScroll: true });
      return;
    default:
      return;
  }

  ev.preventDefault();
  // Roving tabindex: former current becomes -1, next becomes 0
  nodes.forEach((n, i) => n.setAttribute('tabindex', i === nextIdx ? '0' : '-1'));
  nodes[nextIdx].focus();
}

// Attach once on the nav element (event delegation):
nav.addEventListener('keydown', handleArrowNav);

// Initial state: the first "interactive" stage (completed / current / previously-visited) is tabindex="0"
// and all others (including never-visited future stages) are tabindex="-1". Arrow keys rove through ALL
// nodes — including future stages — so keyboard users can inspect every stage's tooltip. Enter on a
// future stage is a no-op because of aria-disabled="true"; the tooltip continues to render on focus.

Tooltip on focus (WCAG 1.4.13 Content on Hover)

  • Hoverable: tooltip is a sibling of the node, not inside it, so pointer can enter the tooltip without dismissing.
  • Dismissible: Escape dismisses without moving focus (see algorithm above).
  • Persistent: the tooltip stays visible as long as focus is on the stage node AND the pointer is not moving to a different element. The CSS above (.stage-node:focus .stage-tooltip, .stage-node:focus-visible .stage-tooltip) guarantees this.
  • Tooltip content is not hover-only: focusing a future stage via arrow keys reveals the SAME tooltip text that hover reveals. No information is gated behind a pointer.

Skip-link z-index verification

The skip link at the top of this document uses focus-visible:z-[100]; the stage nodes' focus outline sits at the element's natural stacking order (no explicit z-*). When the skip link is focused on page load and the first stage simultaneously has tabindex="0", the skip link's z-100 + fixed top-2 left-2 positioning places it at the top-left corner — above and to the LEFT of the first stage node's 2px teal focus ring (which cannot bleed past its own 4px offset). Visual verification is in the Arrow-nav mockup below; no overlap.

Focus-order policy

Every stage node has an explicit tabindex. This eliminates ambiguity between "focusable-but-no-action" and "not in the tab order" — two states that look identical to pointer users but are wildly different for keyboard users.

Stage condition tabindex Hover Focus ring Activation
Completed (prior stage) 0 teal border + tooltip teal 2px outline, 2px offset Enter opens stage (jump-to-stage)
Current (in-progress) 0 diamond badge lifts teal outline on diamond Enter scrolls to stage's active unit
Previously visited (now "future") 0 border darkens, tooltip teal outline Enter opens read-only view of prior visit
Future, never visited -1 tooltip only (no border change) teal outline on focus (FB-65 — reachable via Arrow keys) n/a — no action (aria-disabled)

Future never-visited stages are tabindex="-1" so they are skipped by the Tab key — but the arrow-key roving handler above (FB-65) moves focus to them, which reveals the same tooltip content that hover shows and satisfies WCAG 2.1.1 Keyboard + 1.4.13 Content on Hover. They remain in the a11y tree (role="link" + aria-disabled="true") so assistive tech announces their existence; Enter / Space is a no-op because there is nothing to activate yet.

Touch-target hit-zone (FB-07)

Every stage node has a ≥ 44×44 CSS-px effective hit area per DESIGN-BRIEF §4 "Touch-target rule (hard floor, FB-64)" and WCAG 2.5.5 (AAA) / 2.5.8 (AA, WCAG 2.2). Stage-progress nodes are discrete activatable navigation targets and are explicitly excluded from the WCAG inline-text exception.

All-completed variant — every stage column w-5 h-5 (20×20) Same nav-scoped rule as Mobile 44×44 (✅ WCAG 2.5.5/2.5.8)

The 44×44 zone is a transparent ::before pseudo-element — it does not alter the visible geometry of the circle or diamond. Inner children are lifted to position: relative; z-index: 1; so they paint above the hit-zone layer. min-width: 44px on each stage column also guarantees flex layout reserves ≥ 44px of horizontal real estate per node, so adjacent ::before hit zones never overlap and create ambiguous target assignment.
