# Fix FB-21 — Tactical Plan (planner, bolt 1)

**Finding:** `Review SPA bundle not minified — 919 KB gzipped, 84% over spec ceiling`.
**Feedback:** `.haiku/intents/universal-feedback-model-and-review-recovery/stages/development/feedback/21-review-spa-bundle-not-minified-919-kb-gzipped-84-over-spec-c.md`

## TL;DR

`packages/haiku-ui/vite.config.ts:28` explicitly sets `minify: false` with
no documented rationale. Flip to `minify: "esbuild"` (Vite default), rebuild,
re-run `audit-bundle-size.mjs --update-baseline` to capture the new
`budget-baseline.json`, and lower `budget.json#bundleGzipMaxBytes` to match
the new reality. That's the entire fix for FB-21.

FB-05 (500 KB unit-03 spec ceiling is unmeetable without vendor-splitting)
was already **rejected** by the human — the 1 MB ceiling is the accepted
number. FB-21 is NOT asking to re-litigate that. FB-21 is asking for the
free config-flip win that was left on the table: esbuild minification
typically reclaims 40–60% of un-minified JS before gzip, which is the
strongest single lever available without touching dependencies or the
MCP single-HTML-blob embedder invariant.

## Root cause

`packages/haiku-ui/vite.config.ts:26-39` has:

```ts
build: {
  // Inline everything into a single HTML file
  minify: false,
  sourcemap: true,
  cssCodeSplit: false,
  assetsInlineLimit: Number.POSITIVE_INFINITY,
  rollupOptions: {
    output: {
      manualChunks: undefined,
      inlineDynamicImports: true,
    },
  },
},
```

`minify: false` is orthogonal to every other flag in that block:
- `cssCodeSplit: false` — required for single-HTML-blob.
- `assetsInlineLimit: Number.POSITIVE_INFINITY` — required for single-HTML-blob.
- `manualChunks: undefined` + `inlineDynamicImports: true` — required for the
  MCP embedder (it inlines one JS chunk into one HTML).
- `sourcemap: true` — fine; external `.map` file, not embedded in bundle.
- `minify: false` — **not required by any of the above.** Minification
  produces a single minified JS chunk; the embedder does not care whether
  the chunk is minified or not. Sourcemaps still work (external `.map` file).

There is no comment or tactical-plan entry explaining why `minify: false`
was set. Most likely inherited from a debug/development config during
unit-03 scaffolding. Leaving it off ships ~5 MB of raw JS (confirmed via
`ls -la packages/haiku-ui/dist/assets/` → `4,999,699 bytes` of
`index-noEywxZ6.js`) which gzips to 919 KB.

## Fix approach — one-line config change + baseline recapture

1. **`packages/haiku-ui/vite.config.ts:28`** — change `minify: false` to
   `minify: "esbuild"`. Keep the `// Inline everything into a single HTML file`
   comment (still accurate). Leave every other build-option untouched.
2. **`npm run build -w haiku-ui`** — regenerates `dist/index.html` +
   `dist/assets/*`. Should complete without warnings; esbuild minify is
   Vite's default and requires no additional devDep.
3. **`node packages/haiku-ui/scripts/audit-bundle-size.mjs --update-baseline`**
   — writes the new `budget-baseline.json` with the minified gzip size.
   This is the ONLY path the audit script allows for baseline writes
   (it refuses to overwrite without the flag). The updated baseline is
   intentionally visible in the diff so the reviewer can see the reduction.
4. **`packages/haiku-ui/budget.json`** — lower `bundleGzipMaxBytes` from
   `1048576` (1024 KB) to a new realistic ceiling based on the measured
   minified size. Target: `Math.ceil(newGzipBytes * 1.10 / 1024) * 1024`
   (round up to nearest KB, with 10% headroom for small future deps). If
   minified gzip lands near the unit-03 spec's 500 KB target, tighten the
   ceiling to `512 KB` to codify the win. If it lands in the 550-700 KB
   range, pick the next round number up. Do NOT leave the ceiling at
   1024 KB — the whole point of FB-21 is to capture the reduction into
   a harder budget.
5. **`packages/haiku-ui/budget.json#notes`** — rewrite the `notes` array
   to reflect the new reality: minification enabled, previous 1 MB ceiling
   was a papering-over, current ceiling is based on the measured minified
   blob. Keep the FB-05 context (mermaid + @xyflow/react + elkjs +
   react-markdown + remark still all-in-one-chunk) as the remaining lever
   for a future units, but do not conflate.
6. **Also update the inlined-HTML bundle check downstream** — `packages/haiku/scripts/bundle-haiku-ui.mjs`
   reads the same `budget.json#bundleGzipMaxBytes`, so lowering it auto-
   tightens that enforcement too. No separate change needed.

## Risk assessment

- **Sourcemap still works?** Yes. Vite emits external `.map` files by
  default; minification applies to the `.js` payload but the sourcemap
  maps minified positions back to source. The `sourcemap: true` flag is
  compatible with every `minify` value. Verify by confirming
  `dist/assets/index-*.js.map` still exists after rebuild.
- **MCP embedder breakage?** No. The embedder (`packages/haiku/scripts/bundle-haiku-ui.mjs`
  via `audit-bundle-size.mjs`'s `loadInlinedHtml()`) inlines the contents
  of `/assets/*.js` into `<script type="module">…</script>` and
  `/assets/*.css` into `<style>…</style>`. Minified JS and minified CSS
  inline identically to un-minified; no escaping concern (esbuild never
  emits `</script>` or `</style>` in its output).
- **Parallel-chain clobber risk?** Moderate. `budget.json` and
  `budget-baseline.json` are hot files — other fix-loop chains that
  touch bundle-size (FB-02, FB-05 rejected, possibly FB-03 / FB-04)
  may be editing them in parallel. Builder must `git status` and
  re-read `budget.json` immediately before overwriting. If another chain
  has already minified + updated the baseline, the builder should detect
  that via (c) above (Δ near 0%) and skip to the ceiling-lowering step.
  If the ceiling has already been lowered to match a minified baseline,
  FB-21 is already fixed and this bolt is a verification no-op.
- **Does esbuild-minify break React 19 / @xyflow / mermaid / react-markdown?**
  No. esbuild is the default minifier for `vite@6` (Vite 3+) — it's what
  every other Vite production build in the wild uses. React 19, React DOM,
  @xyflow/react, react-markdown, remark, remark-gfm, remark-html, zod,
  react-window, @tailwindcss/vite, axe-core — all have documented compat
  with esbuild minification. `elkjs` ships as a pre-bundled UMD but
  esbuild minify on a pre-bundled UMD is a no-op at worst, not a breakage.
- **Test-coverage impact?** None. The unit-03 test suite asserts on
  DOM semantics (role/label/text) rendered at runtime. Minification
  preserves runtime behavior exactly. The a11y-pages spec, contrast
  audit, keyboard-shortcut audit, live-region audit, touch-target
  audit — none of them read the minified JS source; they run the app.

## Out of scope

- **Vendor-splitting @xyflow/react + elkjs + mermaid + react-markdown + remark**
  behind a dynamic import. This is the FB-05 territory that was already
  **rejected** — the user accepted the 1 MB ceiling as reasonable at that
  layer of investment. FB-21's scope is bounded to the free config-flip win,
  explicitly NOT a dependency refactor.
- **Lifting the `inlineDynamicImports: true` + `manualChunks: undefined`
  pair.** That's an MCP-embedder invariant. Do not touch.
- **Changing the audit-bundle-size.mjs baseline mechanism** (the 5%
  regression guard is appropriate; the explicit-PR baseline-update
  requirement is appropriate).
- **Updating the website or paper** to reflect minification. Build
  configuration is an implementation concern; sync discipline does not
  apply.
- **Re-litigating FB-02** (builder silently raised the budget) or
  **FB-05** (unit-03 spec ceiling is unmeetable) — both already adjudicated.
  The tighter ceiling FB-21 enables is a net improvement over both.
